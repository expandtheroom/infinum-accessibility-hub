Checkboxes & Radios

Accessible patterns for checkbox and radio button groups, including correct labeling, grouping, and keyboard behavior.

Design guidelines

UX guidelines

  • Labels must always be visible and adjacent to the control — place them to the right of the checkbox or radio, never above or below in isolation. A label that is spatially distant from its control creates confusion for users with low vision or cognitive disabilities.
  • Annotate required groups in design files — indicate which fieldsets require at least one selection, and specify that the requirement indicator (*) must be accompanied by a visible legend explaining its meaning (e.g. "* Required field").
  • Never use a single radio button in isolation. A radio implies mutual exclusivity within a group; a single radio with no alternatives conveys nothing meaningful. Use a checkbox for a single binary choice, or provide at least two radio options.

Visual design guidelines

  • Custom-styled checkboxes and radios must retain all visual states: unchecked, checked, indeterminate (checkbox only), focus, hover, and disabled. When replacing native controls with custom CSS, each state must be explicitly designed — browser defaults no longer apply.
  • Error states must combine color with a secondary indicator — a red border or red text alone fails WCAG 1.4.1 (Use of Color). Add an error icon, an "Error" text label, or an inline error message to communicate the failure without relying on color alone.
  • Touch targets must be at least 44×44px including the label's click area, not just the control itself. Extend the clickable area to the full label text — this is especially important on mobile where precise tapping of small controls is difficult.
  • The indeterminate state must be visually distinct from both checked and unchecked — typically shown as a dash or minus sign inside the checkbox. Do not rely solely on the browser's native rendering; design a custom indeterminate state that works across your component library.

Dev guidelines for web

  • Use <input type="checkbox"> or <input type="radio"> with an associated <label> via matching for/id attributes. Wrapping the input inside the label is an equally valid approach. Never use custom elements with role="checkbox" when a native input will do.
  • Wrap related groups in <fieldset><legend>. Screen readers announce the legend text before each option — e.g. "Preferred contact method, group. Email, radio button, 1 of 3." Without a fieldset and legend, users lose the question context when navigating between options.
  • Indicate required groups using aria-required="true" on at least one input in the group, or on the <fieldset> element itself. Do not rely on an asterisk alone — pair it with a visible legend explaining the convention.
  • Set the indeterminate state via JavaScript: checkboxElement.indeterminate = true. Screen readers announce this as "mixed". Do not use aria-checked="mixed" on a native <input type="checkbox"> — the JavaScript property is the correct mechanism for native inputs.
  • Radio groups use roving tab index — only the selected radio (or the first if none selected) should be in the tab order (tabindex="0"); all others are tabindex="-1". Arrow Up / Arrow Down move focus and selection within the group. Do not make all radios individually tabbable — it creates an unnecessarily long tab sequence.
  • Checkboxes are individually tabbable — each checkbox in a group receives its own Tab stop. Space toggles the checked state. Unlike radio buttons, checking one checkbox does not affect the others.

Example code

<!-- Checkbox group -->
<fieldset>
  <legend>Preferred contact methods</legend>
  <label>
    <input type="checkbox" name="contact" value="email" />
    Email
  </label>
  <label>
    <input type="checkbox" name="contact" value="sms" />
    SMS
  </label>
  <label>
    <input type="checkbox" name="contact" value="phone" />
    Phone
  </label>
</fieldset>

<!-- Radio group -->
<fieldset>
  <legend>
    Subscription plan
    <span aria-hidden="true">*</span>
  </legend>
  <label>
    <input type="radio" name="plan" value="free" />
    Free
  </label>
  <label>
    <input type="radio" name="plan" value="pro" />
    Pro
  </label>
</fieldset>
Preferred contact methods
Subscription plan *
