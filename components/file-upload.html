File Upload Fields

Accessible file upload patterns that work for all users, including keyboard users and those using assistive technologies.

Design guidelines

UX guidelines

  • State accepted file types and size limits in the label or hint text — never leave users to discover restrictions through error messages after a failed upload. Display this information visibly below the label at all times: e.g. "Accepted: PDF, DOCX. Maximum: 5MB."
  • A fallback button must always be visible alongside any drag-and-drop zone — label it "Choose file" or "Browse files". Never provide drag-and-drop as the only upload method. The button must be prominently placed, not hidden below the fold or styled as a secondary action.
  • Annotate upload states in design files: default, drag-hover (file is being dragged over the zone), uploading, success, and error. Each state must be designed and handed off explicitly so developers know what to implement for each transition.

Visual design guidelines

  • Design a visible progress indicator for uploads that take time — a progress bar with a percentage or animated indicator, plus a text status such as "Uploading… 60%". Progress must not rely on color or animation alone to convey state.
  • Error states must combine color with a secondary indicator — a red border plus an error icon and a descriptive message. Write specific errors: "File too large: your file is 8MB, the limit is 5MB" rather than just "Upload failed".
  • Success states must confirm the file name — after upload, display the selected or uploaded filename visibly so users can verify the correct file was chosen. Add a success icon and confirmation message alongside it.
  • Drag-and-drop zones must have a clearly visible boundary — use a dashed or solid border with sufficient contrast (3:1 minimum against the page background) and a label inside the zone describing the action: "Drag files here or…"

Dev guidelines for web

  • Use <input type="file"> with an associated <label> via for/id. The native element opens the file browser dialog when activated by keyboard or pointer. Use the accept attribute to restrict file types: e.g. accept=".pdf,.docx".
  • Link format and size hints via aria-describedby: <input type="file" id="cv" aria-describedby="cv-hint" /> with <p id="cv-hint">Accepted: PDF, DOCX. Maximum: 5MB.</p>. Screen readers announce the hint after the label when the field receives focus.
  • To create a custom-styled upload button, visually hide the native input using a .visually-hidden CSS class (not display:none or visibility:hidden, which remove it from keyboard access), then trigger it from a styled <button> using inputElement.click(), or use a styled <label for="..."> as the visible trigger.
  • Use the multiple attribute when multi-file upload is supported: <input type="file" multiple />. Communicate this affordance in the hint text and design.
  • On validation failure, set aria-invalid="true" on the input and add the error message's id to aria-describedby. The error must be visible and specific: "File too large: 8MB uploaded, 5MB maximum."
  • For async uploads, announce completion or failure using an aria-live="polite" region: e.g. "profile-photo.jpg uploaded successfully." Move focus to the success/error message if the upload result requires user action.
  • Display the selected filename in visible text adjacent to the input after selection. The native input does not display the filename in a consistently accessible way — add a visible text element that updates via JavaScript with the selected file's name.

Example code

<div>
  <label for="resume-upload">
    Upload your resume
    <span aria-hidden="true">*</span>
    <span class="visually-hidden">(required)</span>
  </label>
  <input
    type="file"
    id="resume-upload"
    name="resume"
    accept=".pdf,.doc,.docx"
    required
    aria-describedby="resume-hint"
  />
  <p id="resume-hint">
    Accepted formats: PDF, DOC, DOCX. Maximum size: 5 MB.
  </p>
</div>

Accepted formats: PDF, DOC, DOCX. Maximum size: 5 MB.
