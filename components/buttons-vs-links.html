Buttons vs Links

Buttons and links are often used interachangeably, but they have very different meanings from an accessibility perspective. Regardless of that they look like visually, they should be coded as such:

  • A button is a control that performs an action on the current page — e.g. "Submit form", "Open menu", "Delete item".
  • A link is a navigational element that takes the user to a different page or section — e.g. "View order history", "Read full article". Using the correct element for each purpose is essential for accessibility, as assistive technologies rely on these semantics to provide the right interactions and feedback to users.

Design guidelines

UX guidelines

  • Icon-only buttons must have a visible tooltip on hover and focus showing the button's text label. In design files, annotate icon-only buttons with their accessible name so developers know what text to expose to assistive technology.
  • Use verb-first, action-oriented microcopy on buttons: "Save changes", "Delete account", "Upload photo". Link labels should describe the destination: "View order history", "Read full article". Label clarity is an accessibility requirement, not just a copywriting preference. Avoid "click here" links or repetitive "Learn more" without context.
  • Annotate the semantic role in design files — mark each interactive element as <button> or <a> in your handoff notes. This removes ambiguity for developers and prevents the most common element-swapping mistakes.

Visual design guidelines

  • Minimum touch target size of 44×44px for all buttons. On mobile, small icon buttons are a common failure. Ensure the interactive hit area meets this minimum even if the visible element is smaller, using padding or a transparent overlay.
  • Design all button states explicitly: default, hover, focus, active (pressed), disabled, and loading. Each state must be visually distinct without relying on color alone — use shape, border, opacity, or label changes in addition to color.
  • Destructive buttons (delete, remove, cancel subscription) should use a distinct visual treatment — typically a high-contrast red or warning color — and be spatially separated from primary actions to prevent accidental activation.

Dev guidelines for web

  • Use <button> for actions and <a href> for navigation — never swap them. Buttons are activated with both Enter and Space; links are activated with Enter only. Links support right-click to open in a new tab and update browser history. Swapping elements breaks these built-in behaviors for keyboard and screen reader users.
  • Never use <div> or <span> with an onclick handler as a substitute for a button. These elements have no role, are not keyboard focusable, and receive no keyboard events. A real <button> provides all of this for free.
  • Use the disabled attribute — not just CSS styling — to disable buttons. The disabled attribute removes the element from the tab order and prevents activation. If you need a disabled button to remain focusable (to show a tooltip explaining why it is disabled), use aria-disabled="true" and intercept click/keyboard events manually.
  • Use aria-pressed="true/false" on toggle buttons (e.g. a mute button, a bold toggle in a text editor). This communicates the button's pressed state to screen readers. The button label should remain constant; only the aria-pressed value changes.
  • For loading states, add aria-busy="true" to the button and use a live region to announce completion. Consider also adding aria-label to update the button's accessible name — e.g. from "Submit" to "Submitting…" — so screen reader users understand the in-progress state.
  • Icon-only buttons require an accessible name via aria-label (e.g. aria-label="Close dialog") or a visually hidden <span>. The decorative icon itself should have aria-hidden="true" to prevent it being announced redundantly.
  • An <a> element must always have an href attribute. An anchor without href is not in the tab order, loses its link semantics, and cannot be activated by keyboard. If you need a clickable element that performs an action rather than navigating, use <button>.

Example code

<!-- Use a button for actions -->
<button type="button" onclick="handleSave()">
  Save changes
</button>

<!-- Use a link for navigation -->
<a href="/dashboard">Go to dashboard</a>

<!-- Icon-only button: label describes the action -->
<button type="button" aria-label="Delete item: Report Q4">
  <span aria-hidden="true">&#128465;</span>
</button>

<!-- Never use a link as a button -->
<!-- <a href="#" onclick="doSomething()">Save</a> &#x2718; -->
