Banners

Accessible banner and site-wide alert patterns with correct roles and dismissal behavior.

Design guidelines

UX guidelines

  • Indicate severity using color + icon + text label together — never color alone. A red error banner must also show a recognisable icon and a label such as "Error:" so users who cannot perceive color receive the same information (WCAG 1.4.1).
  • Use a consistent icon set for the four states — info, warning, error, and success — so users learn to recognize severity at a glance.
  • Banners should span the full width of the page or content area, positioned immediately below the site header or at the top of the relevant content region.
  • Avoid auto-dismissing critical banners. If a banner does auto-dismiss, provide at least 5 seconds of reading time and pause the timer when the banner receives keyboard focus or mouse hover.

Visual design guidelines

  • Every dismissible banner must include a clearly visible close button with an adequate touch target (minimum 44×44 px).
  • Ensure sufficient color contrast on all colored banner backgrounds — body text must meet 4.5:1 and large/bold text 3:1 against the banner's background color.
  • Entry animations must respect the prefers-reduced-motion media query — use a static or fade-only alternative rather than a sliding animation for users who have opted into reduced motion.

Dev guidelines for web

  • Use role="alert" for urgent banners (errors, session expiry warnings) — this maps to aria-live="assertive" and interrupts whatever the screen reader is currently announcing.
  • Use role="status" or aria-live="polite" for informational, non-urgent banners — the announcement waits for current speech to finish.
  • Inject message text dynamically into a pre-existing live region container. Do not create and destroy the container on each banner — render it empty on page load and update its text content via JavaScript.
  • Add aria-atomic="true" to the live region so the entire message is read as a single unit rather than only the changed portion.
  • Mark status icons as aria-hidden="true" when a visible severity label is already present in the text, to prevent double-announcing the type.
  • The close button must be a real <button> with a descriptive accessible name: <button aria-label="Dismiss banner">. A bare × glyph with no label is not accessible.
  • Do not move focus to the banner unless it requires immediate user action. On dismissal, return focus to a logical location — the triggering element or the top of the main content area.

Example code

<!-- Informational banner -->
<div role="region" aria-label="Announcement">
  <p>
    <strong>Scheduled maintenance:</strong>
    The site will be unavailable on Sunday, 2 March from
    02:00&#x2013;04:00 UTC.
  </p>
</div>

<!-- Error banner (live region) -->
<div role="alert">
  <strong>Error:</strong> Your session has expired.
  <a href="/login">Sign in again</a>.
</div>
Scheduled maintenance: The site will be unavailable on Sunday, 2 March from 02:00–04:00 UTC.
Warning: Your password expires in 3 days.
Error: Your session has expired. Sign in again.
