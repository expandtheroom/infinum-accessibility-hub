Text Inputs

Accessible text input patterns covering labels, hints, error messages, and assistive technology support.

Design guidelines

UX guidelines

  • The label must always be visible above the input — never replace it with placeholder text alone, never use floating labels that disappear into the border on focus. A persistent, clearly readable label is required by WCAG 1.3.1 (Info and Relationships) and helps users with short-term memory difficulties who cannot re-read a disappeared placeholder while typing.
  • Placeholder text is supplementary only — use it to show an example value (e.g. "e.g. name@example.com") but never as the field's only label. Browser default placeholder contrast fails WCAG 1.4.3; always check and override it to meet at least 4.5:1.
  • Hint text must appear below the label and above any error message — provide format guidance (e.g. "DD/MM/YYYY"), character limits, or other constraints as persistent visible text, not only as placeholder or tooltip text. Position it consistently across the form.
  • Required fields must be indicated with an asterisk (*) accompanied by a visible legend at the top of the form explaining "* Required". Do not use "Optional" labels on every optional field (this becomes noisy); mark required fields and leave optional ones unmarked unless the form is mostly optional.
  • Input width should be proportional to the expected content length — a postcode field should be short; a "Full name" field should be wider. This acts as a visual affordance for how much text is expected, reducing cognitive load.

Visual design guidelines

  • Error messages must appear below the input in a consistent location — use a combination of a red/error color, an error icon, and a descriptive text message. Never rely on color alone (WCAG 1.4.1). The error message must remain visible until the error is resolved.
  • Design a clearly visible focus ring — specify the focus style in design files (color, width, offset). Do not leave it to browser defaults, which may be invisible in certain design contexts. Annotate that outline: 0 must never be used without a replacement.

Dev guidelines for web

  • Associate every input with a <label> via matching for/id attributes. Clicking the label moves focus to the input — an important usability benefit, especially for motor-impaired users on touch devices. Never use aria-label as a substitute when a visible label can be provided.
  • Link hint text via aria-describedby: <input type="text" id="phone" aria-describedby="phone-hint" /> with <p id="phone-hint">Include your country code, e.g. +44 7700 900000</p>. Screen readers announce the hint after the label when the user focuses the field.
  • On validation failure, set aria-invalid="true" on the input and add the error message element's id to aria-describedby (space-separated alongside any existing hint id). Remove aria-invalid or set it to "false" once corrected. Screen readers announce "invalid entry" alongside the label and role.
  • Use the appropriate type attribute — type="email", type="tel", type="url", type="number" — to trigger the correct on-screen keyboard on mobile and to enable browser-level validation. Use type="text" only when no more specific type applies.
  • Provide autocomplete attributes for personal data fields — e.g. autocomplete="name", autocomplete="email", autocomplete="tel", autocomplete="street-address". This enables browsers and password managers to prefill known values, which is essential for users with motor disabilities or cognitive impairments.
  • Never use outline: 0 or outline: none without providing a custom, clearly visible focus replacement. WCAG 2.2 SC 2.4.11 (Focus Appearance) requires a minimum area and contrast ratio for focus indicators. Specify your custom focus style explicitly in CSS.
  • Input borders must meet 3:1 contrast against the adjacent background — a common failure is very light grey borders on a white page. Input text must meet 4.5:1. Mark both requirements in your design system tokens to make them easy to audit.

Example code

<div>
  <label for="email-input">
    Email address
    <span aria-hidden="true">*</span>
    <span class="visually-hidden">(required)</span>
  </label>
  <input
    type="email"
    id="email-input"
    name="email"
    required
    autocomplete="email"
    aria-describedby="email-hint"
  />
  <p id="email-hint">We'll use this to send your receipt.</p>
</div>

<!-- Invalid state -->
<div>
  <label for="email-error">Email address</label>
  <input
    type="email"
    id="email-error"
    aria-invalid="true"
    aria-describedby="email-error-msg"
    value="not-an-email"
  />
  <p id="email-error-msg" role="alert">
    Please enter a valid email address.
  </p>
</div>

We'll use this to send your receipt.

⚠ Please enter a valid email address.
