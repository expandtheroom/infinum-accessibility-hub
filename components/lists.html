Lists

Accessible list patterns using correct semantic HTML for ordered, unordered, and description lists.

Design guidelines

UX guidelines

  • If content looks like a list (parallel items, consistent structure) it should be marked up as a list — do not style a list to look like a paragraph to avoid bullet points.
  • Keep nesting to a maximum of three levels to avoid cognitive overload; deep nesting is a signal to reconsider the information architecture.

Visual design guidelines

  • Use consistent indentation for nested lists so the hierarchy is visually clear at a glance; each level should be distinctly indented relative to its parent.
  • List items should have adequate spacing between them — cramped items are harder to read for users with cognitive or visual impairments.
  • Ordered list numbers must be legible — ensure sufficient size, contrast, and that they are not accidentally hidden by list-style: none when order matters.
  • Definition lists (<dl>) should use a consistent visual pairing — e.g. term bold above definition — so the term/value relationship is immediately apparent.

Dev guidelines for web

  • Use <ul> for unordered items, <ol> for ordered sequences, and <dl><dt><dd> for term/value pairs such as metadata, glossaries, and FAQs.
  • Nested lists must be placed inside the relevant <li> element — not as a sibling after it — to maintain correct DOM structure and screen reader announcements.
  • Applying list-style: none in CSS causes Safari/VoiceOver to strip list semantics. Add role="list" on <ul>/<ol> to restore the "list, N items" announcement when markers are intentionally hidden.
  • Screen readers announce "list, 5 items" — this count is a valuable orientation cue that helps users decide whether to explore or skip the list.
  • Do not use <ul> or <ol> for visual indentation or layout with no genuine list relationship; this misleads screen reader users who expect related peer items.
  • <dl><dt><dd> is the correct pattern for key/value data — it provides a programmatic term/definition association that prose paragraphs do not.

Example code

<!-- Unordered list -->
<ul>
  <li>Use semantic HTML before reaching for ARIA</li>
  <li>Test with real screen readers</li>
  <li>Ensure keyboard navigability</li>
</ul>

<!-- Ordered list -->
<ol>
  <li>Open the accessibility panel</li>
  <li>Run an automated audit</li>
  <li>Review flagged issues</li>
</ol>

<!-- Description list for key-value pairs -->
<dl>
  <dt>WCAG version</dt>
  <dd>2.2</dd>
  <dt>Conformance level</dt>
  <dd>AA</dd>
</dl>

Unordered

  • Use semantic HTML before ARIA
  • Test with real screen readers
  • Ensure keyboard navigability

Ordered

  1. Open the accessibility panel
  2. Run an automated audit
  3. Review flagged issues

Description list

WCAG version
2.2
Conformance level
AA
