Modals & Dialogs

Accessible modal and dialog patterns with focus trapping, keyboard management, and correct ARIA roles.

Design guidelines

UX guidelines

  • Always include a visible heading in the modal — it gives users an immediate understanding of what the dialog is for and is required for the programmatic accessible name.
  • Place primary and secondary actions in the modal footer, clearly labeled. Cancel should always be available to give users an escape hatch.
  • Avoid nested modals — they dramatically increase cognitive and interaction complexity for all users.

Visual design guidelines

  • Include a close button (×) in the top-right corner with an adequate touch and click target of at least 44×44 px.
  • Use a dimmed backdrop to communicate that background content is temporarily inactive.
  • Modal width should be responsive with a sensible max-width (around 600 px for forms). The modal itself should not be wider than the viewport on small screens.
  • Content that overflows the modal height should scroll within the modal — do not scroll the whole page behind it.

Dev guidelines for web

  • Use role="dialog" with aria-modal="true" and aria-labeledby pointing to the dialog's visible heading: <div role="dialog" aria-modal="true" aria-labelledby="dialog-title">. Prefer the native <dialog> element where browser support allows — it handles much of this natively.
  • Move focus into the dialog on open — to the container (tabindex="-1"), the heading, or the first focusable element. Screen reader users will not know the dialog has appeared unless focus is moved inside it.
  • Trap focus within the dialog: Tab on the last focusable element cycles to the first; Shift+Tab from the first cycles to the last.
  • Support Escape to close — implement a keydown listener on the dialog that fires regardless of which element currently has focus.
  • Apply the inert attribute to all content outside the dialog while it is open (e.g. document.getElementById('app').inert = true). Remove it on close. Do not rely on aria-modal="true" alone — cross-browser and cross-screen-reader support is inconsistent.
  • Return focus to the trigger on close — store a reference to the trigger before opening and call trigger.focus() when the dialog closes, via any mechanism.

Example code

<button
  type="button"
  onclick="document.getElementById('demo-modal').removeAttribute('hidden');
           document.getElementById('modal-close').focus();"
>
  Open dialog
</button>

<div
  role="dialog"
  id="demo-modal"
  aria-modal="true"
  aria-labelledby="modal-title"
  aria-describedby="modal-desc"
  hidden
>
  <h2 id="modal-title">Confirm deletion</h2>
  <p id="modal-desc">
    This action cannot be undone.
    The item will be permanently deleted.
  </p>
  <button type="button">Delete permanently</button>
  <button
    type="button"
    id="modal-close"
    aria-label="Cancel and close dialog"
    onclick="document.getElementById('demo-modal').hidden=true;"
  >
    Cancel
  </button>
</div>

Confirm deletion
