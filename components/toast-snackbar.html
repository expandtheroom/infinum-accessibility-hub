Toast / Snackbar Messages

Accessible toast and snackbar patterns using ARIA live regions for transient announcements.

Design guidelines

UX guidelines

  • Toasts should appear in a consistent position across the app — typically bottom-center or top-right — so users know where to look and can anticipate them.
  • Indicate severity using color + icon + text together — never color alone. Include the severity in the message: e.g. "Success: File uploaded" or "Error: Unable to save changes."
  • Auto-dismiss duration must be at least 5 seconds for short messages (WCAG 2.2.1). Pause the timer when the toast receives keyboard focus or mouse hover so users with slower reading speeds have time to read it.
  • Always include a manual dismiss button so users are not forced to wait for auto-dismissal. This is especially important for keyboard users.
  • Never put critical, blocking errors in a toast. Errors that prevent task completion must be persistent and visible — use an inline error or a persistent banner instead.

Visual design guidelines

  • Ensure sufficient color contrast on all toast background colors — body text must meet 4.5:1 against the toast background.
  • Stacked toasts must be clearly separated and must not overlap each other or important page content.

Dev guidelines for web

  • Render an aria-live="polite" region in the DOM on page load, before any toasts fire: <div aria-live="polite" aria-atomic="true" class="toast-region"></div>. Inject toast text into it dynamically — do not create and destroy the container.
  • Use aria-atomic="true" so the entire message is always read as a single unit, not just the changed portion.
  • Never use role="alert" for non-urgent toasts — it asserts itself aggressively. aria-live="polite" is correct for most toast messages.
  • Clear the live region content after the toast is dismissed so it is ready for the next message. Do not leave stale text in the region.
  • The dismiss button must be a real <button> with a descriptive accessible name: <button aria-label="Dismiss">. After dismissal, return focus sensibly — typically to the element the user was interacting with before the toast appeared.
  • If the toast contains an action such as "Undo", it must remain visible long enough for a keyboard user to navigate to it. In practice this is hard to guarantee — prefer placing the action inline on the page or in a persistent banner.
  • Queue toasts and display them sequentially rather than stacking multiple live region announcements simultaneously, which creates a confusing flood of speech for screen reader users.

Example code

<!-- Polite toast -->
<div
  role="status"
  aria-live="polite"
  aria-atomic="true"
  class="toast"
>
  <span aria-hidden="true">&#10003;</span>
  Changes saved successfully.
  <button
    type="button"
    aria-label="Dismiss notification"
    onclick="this.parentElement.hidden = true"
  >
    &#10005;
  </button>
</div>

<!-- Assertive toast for errors -->
<div
  role="alert"
  aria-live="assertive"
  aria-atomic="true"
  class="toast toast--error"
>
  <span aria-hidden="true">&#9888;</span>
  Failed to save. Please try again.
</div>
✓ Changes saved successfully.
⚠ Failed to save. Please try again.
