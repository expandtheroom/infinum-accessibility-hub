Password Fields

Accessible password input patterns covering visibility toggles, error messaging, and screen reader support.

Design guidelines

UX guidelines

  • The show/hide toggle button must be adjacent to the password field with a clear icon (e.g. an eye or eye-slash) plus a visible or screen-reader-accessible text label. Place it consistently in the same position across all password fields in the product to build a predictable pattern.
  • Design password requirements as visible persistent text below the field (e.g. "At least 8 characters, one uppercase letter, one number"), not revealed only on error. This reduces cognitive load for all users and prevents frustrating trial-and-error.
  • Annotate the toggle button's accessible name in design files — specify that the text must change from "Show password" to "Hide password" (or equivalent) to reflect the current state, not just the icon.
  • Consistent toggle placement across all password fields — if the toggle appears inside the input on the right for the first field, it must appear in the same position for confirm-password and new-password fields. Inconsistency creates confusion, particularly for users with cognitive disabilities.

Visual design guidelines

  • Password strength meters must use color plus a text label — never color alone. Display labels such as "Weak", "Fair", "Strong", or "Very strong" alongside the visual bar or segmented indicator. A meter that only changes from red to green fails WCAG 1.4.1 (Use of Color).
  • Strength labels must be visible at all times the meter is displayed — not just on hover, not only announced to screen readers, but rendered as visible text adjacent to the strength bar.

Dev guidelines for web

  • Use <input type="password"> with a visible, associated <label> via for/id. Set autocomplete="current-password" on login forms and autocomplete="new-password" on registration and password-change forms — this allows password managers, which many users with cognitive or motor disabilities rely on, to fill credentials automatically.
  • The show/hide toggle must be a <button type="button"> with aria-pressed="false" (updated to aria-pressed="true" when password is visible). Toggle the input's type attribute between "password" and "text" to control visibility. Update the button's accessible label to reflect the available action: "Show password" or "Hide password".
  • Never block paste on password fields. Intercepting the paste event (e.g. onpaste="return false") is a WCAG 2.1 SC 1.3.5 failure and prevents password managers from filling credentials. Remove any paste-blocking handlers entirely.
  • Never set autocomplete="off" on login forms. This blocks password managers and browser autofill, which are essential assistive tools for users with memory, motor, or cognitive disabilities.
  • Announce strength changes via aria-live="polite": <p aria-live="polite" aria-atomic="true">Password strength: Strong</p>. Use aria-atomic="true" so the full message is announced on each change. Avoid aria-live="assertive" — it is too interruptive during active typing.
  • If the strength meter is a visual bar or segmented graphic, supplement it with a role="meter" element carrying aria-valuenow, aria-valuemin, aria-valuemax, and aria-valuetext (e.g. aria-valuetext="Strong") to expose the value programmatically to assistive technology.
  • Link password requirements to the input via aria-describedby so screen readers announce them when the field receives focus — users hear the requirements before they start typing, not just after a failed submission.

Example code

<div>
  <label for="password">
    Password
    <span aria-hidden="true">*</span>
    <span class="visually-hidden">(required)</span>
  </label>
  <div>
    <input
      type="password"
      id="password"
      name="password"
      required
      autocomplete="current-password"
      aria-describedby="password-hint"
    />
    <button
      type="button"
      aria-controls="password"
      aria-pressed="false"
      onclick="
        var i = document.getElementById('password');
        var p = this.getAttribute('aria-pressed') === 'true';
        i.type = p ? 'password' : 'text';
        this.setAttribute('aria-pressed', !p);
        this.textContent = p ? 'Show' : 'Hide';
      "
    >
      Show
    </button>
  </div>
  <p id="password-hint">Minimum 8 characters.</p>
</div>

Minimum 8 characters.
