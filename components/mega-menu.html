Mega Menu

Accessible mega menu patterns with keyboard navigation, focus management, and ARIA roles.

Design guidelines

UX guidelines

  • The mega menu panel must be visually connected to its trigger — use proximity, a border, or a notch so users understand the relationship between the button and the content below it.
  • Use clear visual grouping within the panel — distinct column headings, adequate whitespace, and logical link ordering so users can scan efficiently.
  • Provide a visible close affordance (×) in the panel so mouse and touch users can dismiss it without moving the cursor out of the panel area.
  • Prefer hover-to-open on desktop, but make top-level items also clickable links. Research shows hover-to-open outperforms click-to-open for desktop navigation efficiency. However, the top-level trigger should also be a navigable link (to the category overview page), not solely a button that opens the panel. Users have different mental models — some expect clicking "Products" to open the submenu, others expect it to navigate. Accommodate both.
  • Repeat the top-level category as a descriptive overview link inside the dropdown. Include an "All [Category]" or "[Category] overview" link as the first item in the panel. This pattern significantly reduces confusion: users who click the top-level link land on the category page, while users who expect a submenu find it. Research found that omitting this link caused visible confusion in 22% of test participants and triggered rage-clicking.
  • Use descriptive labels, not generic ones. "All classes" outperforms "Overview" as an overview link because it carries stronger information scent — users can predict what they will find before clicking.
  • On mobile, the top-level item should open the submenu rather than navigate directly. Mobile users need access to both the parent category page and the submenu. The overview link inside the panel provides the path to the parent page, so the top-level tap can safely open the panel without navigating away.
  • Avoid arrow-only tap targets on mobile. Small chevron buttons beside top-level links are insufficient touch targets and add unnecessary interaction complexity. Use the full top-level item as the tap target to open the submenu.
  • Standardize the interaction pattern across the entire navigation. Inconsistent behavior — some items opening panels, others navigating directly — forces users to re-learn the interface with every item. Consistency reduces cognitive load and prevents lost navigation attempts.

Visual design guidelines

  • Use a backdrop or overlay to dim the page content behind the open panel, reinforcing that the panel is in the foreground.
  • All panel open/close animations must respect prefers-reduced-motion — provide an instant show/hide alternative when the user has reduced motion enabled.
  • Ensure sufficient padding around panel items so links do not feel cramped and touch targets remain large enough.
  • Active and hover states on panel links must be visually clear — not color alone.

Dev guidelines for web

  • Use a split pattern for top-level nav items: an <a> for the label that navigates to the category overview page, and a separate <button> (typically a chevron icon button) for opening the dropdown panel. The button carries aria-expanded and aria-controls; the link carries aria-describedby pointing to the button so screen reader users understand the relationship. This satisfies both navigation and disclosure behaviors without compromise.
  • Each trigger must be a <button aria-expanded="false" aria-controls="panel-id"> — never a <div> or an <a> masquerading as a button.
  • The panel element must have an id matching the trigger's aria-controls value; update aria-expanded to "true" when open and "false" when closed.
  • On open, move focus programmatically to the first interactive item in the panel — do not leave focus on the trigger after the panel opens.
  • Within the panel, support Arrow key navigation: ↓/↑ between items in a column, →/← between columns; use a roving tabindex so only the focused item has tabindex="0".
  • Pressing Escape must close the panel and return focus to the trigger that opened it.
  • Tab from the last item in the panel exits naturally; use display: none or visibility: hidden on the closed panel — not opacity: 0 — to fully remove it from the tab order and accessibility tree.
  • On mobile, apply aria-hidden="true" or the inert attribute to background content when the panel is open as a modal overlay.

Example code

<nav aria-label="Main navigation">
  <ul role="list">
    <li>
      <button
        type="button"
        aria-haspopup="true"
        aria-expanded="false"
        id="menu-products-btn"
        aria-controls="menu-products-panel"
      >
        Products
      </button>
      <div
        id="menu-products-panel"
        role="region"
        aria-labelledby="menu-products-btn"
        hidden
      >
        <ul role="list">
          <li><a href="/products/design">Design tools</a></li>
          <li><a href="/products/dev">Developer tools</a></li>
          <li><a href="/products/audit">Audit tools</a></li>
        </ul>
      </div>
    </li>
  </ul>
</nav>
