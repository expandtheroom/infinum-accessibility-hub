Timelines

Accessible timeline patterns with clear temporal structure and screen reader support.

Design guidelines

UX guidelines

  • Chronological order must be clear from the visual layout — top-to-bottom or left-to-right; do not rely on position alone to imply sequence.
  • Each event must provide sufficient context in its visible content — do not rely on the connector line's position to explain what happened or when.
  • Expandable timeline events must have a clear expand affordance (e.g. a visible toggle button or chevron) — do not hide interactivity behind hover states only.

Visual design guidelines

  • Dates must be prominent and legible — sufficient size and contrast so they are immediately distinguishable from event descriptions.
  • The current or active event must be visually distinct (e.g. highlighted node, stronger color, bold label) — not distinguished by color alone.
  • Connector lines and node dots are decorative — they must not carry meaning not present in the text content of each event.

Dev guidelines for web

  • Use <ol> for chronological timelines — the ordered list announces item count and implies sequence to screen reader users; use <ul> only for thematically ordered (non-chronological) timelines.
  • Wrap each event date in <time datetime="2025-03-01">1 March 2025</time> — the ISO 8601 datetime attribute provides a machine-readable value for assistive technologies and search engines.
  • Decorative connector lines and dot nodes must have aria-hidden="true" (inline SVG or <div>) — CSS-generated ::before/::after content is ignored automatically and needs no ARIA treatment.
  • DOM order must match the visual reading order — never use CSS order or position: absolute to visually reorder events in a way that contradicts the DOM sequence.
  • Expandable events use the accordion pattern: <button aria-expanded="false" aria-controls="panel-id"> inside the <li>; the panel uses hidden to remove it from the accessibility tree when collapsed.
  • Link text in timeline items must be descriptive — include event title and date (e.g. "Beta release — 15 June 2024") — never use "Read more" as standalone link text, which is meaningless out of context for screen reader users navigating by links.

Example code

<section aria-labelledby="timeline-heading">
  <h2 id="timeline-heading">Project history</h2>
  <ol>
    <li>
      <time datetime="2024-03-01">March 2024</time>
      <p>Project kickoff and requirements gathering.</p>
    </li>
    <li>
      <time datetime="2024-06-15">June 2024</time>
      <p>Design system completed and handed off to development.</p>
    </li>
    <li>
      <time datetime="2024-09-30">September 2024</time>
      <p>Accessibility audit: 0 critical issues remaining.</p>
    </li>
  </ol>
</section>

Project history

  1. March 2024

    Project kickoff and requirements gathering.

  2. June 2024

    Design system completed and handed off to development.

  3. September 2024

    Accessibility audit: 0 critical issues remaining.
