Accordions

Accessible accordion patterns with correct ARIA expanded states and keyboard navigation.

Design guidelines

UX guidelines

  • Make the accordion trigger visually look like a button. Do not style the trigger as plain text or a heading alone. It must be visually distinguishable as an interactive control — with a background, border, or clear affordance — so users know it can be clicked or tapped.
  • Where possible, design panels so fully expanded content is visible without additional scroll. Long panels that require scrolling within them are confusing. If panel content is very long, consider whether the accordion is the right pattern.

Visual design guidelines

  • Use a chevron icon to indicate expand/collapse state. The chevron should point down when the panel is collapsed and rotate to point up when it is open. This provides a clear visual indicator of state at a glance.
  • Mark the chevron icon as aria-hidden. The icon is a visual affordance only. Its state is communicated programmatically via aria-expanded on the button, so the icon itself should be hidden from assistive technology to avoid redundant announcements.
  • Use panel borders or shadows to visually connect the open panel to its trigger. A continuous visual boundary helps users understand which trigger controls which content, reducing cognitive effort.
  • Respect prefers-reduced-motion for the expand/collapse animation. If the panel uses a height or opacity transition, disable or reduce the motion when the user has indicated they prefer less animation.

Dev guidelines for web

  • The trigger must be a <button> nested inside a heading: <h3><button aria-expanded="false" aria-controls="panel-id">…</button></h3>. The heading maintains document outline for users who navigate by headings; the button ensures the trigger is keyboard-focusable and announced as interactive.
  • The panel element must have an id, role="region", and aria-labeledby pointing to the button's id. This creates an explicit programmatic relationship between trigger and content that some screen readers expose as a shortcut.
  • Toggle aria-expanded="true" / aria-expanded="false" on the button via JavaScript. Screen readers announce this as "expanded" or "collapsed" when the button receives focus, communicating state without relying on visual cues alone.
  • Hide collapsed panels using the hidden attribute or display: none — not visibility: hidden. visibility: hidden hides the panel visually but may leave interactive elements within it in the tab order, causing users to focus elements they cannot see.
  • Keyboard: Enter or Space toggles the panel; Tab moves into open panel content sequentially. Focus must stay on the trigger button when a panel opens or closes — do not move it automatically.

Example code

<div>
  <h3>
    <button
      type="button"
      aria-expanded="false"
      aria-controls="accordion-panel-1"
      id="accordion-btn-1"
    >
      What is WCAG?
    </button>
  </h3>
  <div
    id="accordion-panel-1"
    role="region"
    aria-labelledby="accordion-btn-1"
    hidden
  >
    <p>WCAG (Web Content Accessibility Guidelines) is a set of
    internationally recognized standards for making web content
    accessible to people with disabilities.</p>
  </div>
</div>

WCAG (Web Content Accessibility Guidelines) is a set of internationally recognized standards for making web content accessible to people with disabilities.
