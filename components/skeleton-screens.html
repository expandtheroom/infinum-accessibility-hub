Skeleton Screens

Accessible skeleton screen patterns that communicate loading state without disrupting the user experience.

Design guidelines

UX guidelines

  • Do not show skeleton screens longer than necessary. If data fails to load, replace the skeleton with a meaningful error message rather than leaving an indefinitely loading placeholder.

Visual design guidelines

  • Skeleton shapes should approximate the real content shapes — a wide rectangle for a heading, narrower lines for body text, a square for an image — to set accurate visual expectations and reduce layout shift when content loads.
  • The shimmer or pulse animation should be subtle. Avoid aggressive motion that could be distracting or cause discomfort.
  • Provide a static skeleton alternative for users with prefers-reduced-motion enabled — use a flat, non-animating placeholder instead of the shimmer animation.
  • The transition from skeleton to real content should feel smooth — avoid jarring layout shifts or a flash of unstyled content appearing abruptly.
  • Skeleton screens should match the consistent visual style of the app — use the same border-radius, spacing, and background tones as the content they represent.

Dev guidelines for web

  • Set aria-hidden="true" on skeleton placeholder elements — or on the skeleton container as a whole — to prevent the screen reader from announcing meaningless blank shapes or empty text nodes.
  • Alternatively, set aria-busy="true" on the loading region: <section aria-busy="true" aria-label="Dashboard loading">. Set it to false only after real content is fully rendered in the DOM.
  • You may combine both approaches: aria-busy="true" on the region (to signal loading is in progress) and aria-hidden="true" on individual skeleton shapes (to suppress meaningless elements from the accessibility tree).
  • Skeleton elements must never be focusable. Do not render placeholder interactive elements such as skeleton buttons or skeleton links — they have no action and no accessible name.
  • Do not apply semantic roles such as role="img" to skeleton shapes — they carry no meaning and the role would be misleading.
  • When real content replaces the skeleton, announce the change via a pre-existing aria-live="polite" region — e.g. inject "Dashboard loaded". If the user navigated explicitly (e.g. submitted a search), move focus to the heading of the loaded content instead of relying solely on the live region.
  • Debounce live region announcements to avoid flooding the user with speech when multiple sections load in rapid succession.

Example code

<!-- Hide skeletons from AT; announce loading state separately -->
<div
  aria-live="polite"
  aria-busy="true"
  aria-label="Loading content"
>
  <div aria-hidden="true" class="skeleton-card">
    <div class="skeleton skeleton--title"></div>
    <div class="skeleton skeleton--line"></div>
    <div class="skeleton skeleton--line skeleton--short"></div>
  </div>
</div>

<!-- Once loaded, replace content and set aria-busy="false" -->
