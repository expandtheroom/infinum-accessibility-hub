Carousels / Sliders

Accessible carousel and slider patterns with pause controls, keyboard navigation, and live region announcements.

Design guidelines

UX guidelines

  • Include a pause/stop button that is always visible. Autoplay must be user-controllable at any time. The pause button should be the first interactive element inside the carousel region and must never be hidden or clipped.
  • Label prev/next controls with text, not just icons. Provide visible text labels or visually hidden supplementary text — for example "Previous slide" / "Next slide" — so users can distinguish controls from one another at a glance.
  • Show a slide position indicator. Display either dot indicators or a text counter such as "2 of 5" so users always know where they are within the sequence.
  • Autoplay should be off by default. Do not assume users want animation. If autoplay must be on, the minimum dwell time per slide is 5 seconds to give users time to read and respond.
  • Question whether a carousel is the right choice. An image grid, a list of featured cards, or a simple section with a heading often gives users better access to the same content without any accessibility burden.

Visual design guidelines

  • Disable autoplay motion when the user prefers it. Respect prefers-reduced-motion by disabling autoplay transitions entirely — not just slowing them down.

Dev guidelines for web

  • Pause/resume button: use aria-pressed or update aria-label to reflect state. When autoplay is active, the button label should read "Pause slideshow"; when paused, "Play slideshow". Toggling aria-pressed is an alternative pattern for a toggle button.
  • Pause autoplay when keyboard focus enters the carousel. Listen for focusin events on the carousel container and suspend rotation so users are not interrupted mid-interaction.
  • Prev/next controls must be <button> elements with descriptive aria-label values. Both must be reachable by Tab and activated by Enter or Space.
  • Use aria-live="polite" on the slide counter or manage announcements with aria-roledescription="carousel". Each slide should carry role="group" and aria-label="Slide 2 of 5" so screen reader users understand position and context.
  • Remove hidden slides from the tab order and accessibility tree. Use the hidden attribute or display: none on non-active slides — not visibility: hidden alone, which leaves content traversable by keyboard while concealing it visually.
  • Honor prefers-reduced-motion by disabling autoplay in CSS and JavaScript. Use a media query to detect the preference and stop the rotation timer on load if it is set.

Example code

<section
  aria-label="Featured articles"
  aria-roledescription="carousel"
>
  <div aria-live="polite" aria-atomic="true">
    Slide 1 of 3
  </div>
  <div
    role="group"
    aria-roledescription="slide"
    aria-label="1 of 3: Intro to WCAG"
  >
    <h3>Intro to WCAG</h3>
    <p>An overview of the Web Content Accessibility Guidelines.</p>
  </div>
  <button type="button" aria-label="Previous slide">&#8249;</button>
  <button type="button" aria-label="Next slide">&#8250;</button>
</section>
