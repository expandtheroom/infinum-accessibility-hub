Toggle Switches

Accessible toggle switch patterns that communicate on/off state clearly to all users.

Design guidelines

UX guidelines

  • "On/Off" or "Enabled/Disabled" text adjacent to or within the toggle helps all users understand the current state at a glance. If space allows, show the current state label next to the thumb or beside the control.
  • Provide immediate visual feedback on toggle activation — the state change must be visible instantly. Never delay the visual state change pending a server response; optimistic UI updates are expected. If the change is undone by a server error, show an error state.
  • The label describes what is being toggled, not its current state — label it "Email notifications", not "Enable email notifications" or "Notifications: On". The state is communicated separately through the visual toggle position and ARIA.
  • Annotate in design files that toggles take immediate effect — distinguish them clearly from checkboxes in a form. Note that no Save button is required, which affects layout and the surrounding component design.

Visual design guidelines

  • On/off state must not rely on color alone — a toggle that is only green when on and grey when off fails WCAG 1.4.1 (Use of Color). Combine color with a position change (thumb moves from left to right), a label change within the toggle ("On"/"Off"), or both, so the state is clear to users who cannot distinguish colors.
  • Touch target must be at least 44×44px — the entire toggle track and surrounding area should be interactive, not just the thumb. On narrow toggles, use padding to extend the hit area.
  • The off state must have sufficient contrast — a very light grey off-state toggle on a white background may fail the 3:1 UI component contrast requirement (WCAG 1.4.11). Ensure the toggle border or fill meets contrast in both on and off states.

Dev guidelines for web

  • Preferred approach: use <input type="checkbox"> styled as a toggle — the native checkbox's checked state maps directly to on/off. It is keyboard operable with Space, announced correctly by screen readers as "checkbox, checked/unchecked", and requires only CSS to restyle. No JavaScript or ARIA needed for core accessibility.
  • Alternative: use role="switch" on a <button> when a native checkbox cannot be used. Set aria-checked="true" when on and aria-checked="false" when off. Update the attribute via JavaScript on activation. Screen readers announce this as "switch, on" or "switch, off".
  • The label must never change — only the ARIA state (aria-checked) or the checkbox's checked property changes. Changing the label text (e.g. from "Enable" to "Disable") simultaneously with the state causes screen readers to announce confusing output where both the name and state change at once.
  • Keyboard interaction: Space toggles the switch state (native checkbox behavior). For role="switch" on a <button>, both Space and Enter should activate it, following button semantics.
  • Toggles must take immediate effect — apply the setting change as soon as the user activates the control. No Save or Submit button should be required. This is the key semantic distinction from a checkbox within a submitted form.
  • Do not group toggles in a <fieldset> unless they collectively form a mutually dependent set. Individual settings toggles in a preferences panel are independent controls; each one has its own label and manages its own state independently.
  • For toggles that affect the page immediately (e.g. dark mode, reduced motion), consider announcing the change with aria-live="polite" so screen reader users are informed of the resulting state: e.g. "Dark mode enabled."

Example code

<!-- Native checkbox styled as a switch -->
<label>
  <input
    type="checkbox"
    role="switch"
    id="dark-mode"
    name="dark-mode"
  />
  Dark mode
</label>

<!-- Button with role="switch" -->
<button
  type="button"
  role="switch"
  aria-checked="false"
  onclick="this.setAttribute('aria-checked',
    this.getAttribute('aria-checked') === 'true'
      ? 'false' : 'true')"
>
  Email notifications
</button>
