Empty States

Accessible empty state patterns that clearly communicate the absence of content to all users.

Design guidelines

UX guidelines

  • The heading must explain what is empty and why — prefer specific copy like "You have no saved reports. Create a report to see it here." over vague text like "Nothing here yet."
  • Illustrations are supplementary, not the sole communicator. Every empty state must include a visible heading and explanatory text that carries all of the meaning independently of any graphic.
  • Include a clear call to action whenever the user can resolve the empty state — for example, a button to create the first item or adjust search filters.
  • The heading level must fit the page outline — usually <h2> if the page already has an <h1>. Do not skip heading levels or default to <h1> inside a subsection.

Visual design guidelines

  • All text must meet sufficient color contrast ratios — 4.5:1 for body text, 3:1 for large/bold text. Low-contrast placeholder or hint text is a common failure on empty states.
  • The empty state must feel intentional, not broken. Use the same typographic scale, spacing, and component style as the rest of the app so it reads as part of the designed experience.
  • Avoid using color or font size alone to distinguish the empty-state heading from body text — use proper semantic heading elements.

Dev guidelines for web

  • Empty state content must be in the normal DOM — never hidden with aria-hidden="true" or display:none. Screen reader users must be able to reach and read it.
  • Decorative illustrations must be marked aria-hidden="true" or given an empty alt="" on an <img>. The adjacent text carries all meaning.
  • The call-to-action must be a proper <button> for actions or an <a> for navigation — never a <div> or <span> with a click handler. Provide a descriptive accessible name; prefer "Create your first report" over a generic "Get started."
  • When an empty state appears dynamically (e.g. after a user deletes all items), move focus to the empty-state heading or announce via an aria-live="polite" region. Do not leave focus on an element that no longer exists.
  • If the empty state is injected into the DOM after a user action, use a role="status" live region to announce its appearance to screen readers that may not notice the DOM change automatically.
  • Heading levels must follow the page outline — typically <h2> if the page has an <h1>. Incorrect heading hierarchy breaks navigation for screen reader users who browse by heading.

Example code

<section aria-labelledby="empty-heading">
  <img
    src="empty-inbox.svg"
    alt=""
    aria-hidden="true"
  />
  <h2 id="empty-heading">No messages yet</h2>
  <p>
    When someone sends you a message, it will appear here.
  </p>
  <a href="/compose">Compose your first message</a>
</section>
