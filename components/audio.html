Audio

Accessible audio patterns with player controls, transcripts, and assistive technology support.

Design guidelines

UX guidelines

  • Place the transcript link directly adjacent to the player. Users should not have to hunt for the text alternative. A clearly labeled link or an inline <details> block immediately below the player is the expected pattern.
  • Never autoplay audio. Autoplay disrupts screen reader speech and can be startling. Audio must not start without an explicit user action.
  • Include speaker labels in multi-speaker recordings. Prefix each contribution with the speaker's name or role, for example HOST: / GUEST:, so the transcript is unambiguous without sound.
  • Timestamps are optional but welcome. Adding timestamps to long transcripts (such as podcasts) helps users jump to relevant sections and cross-reference with the audio.

Visual design guidelines

  • Keep native player controls visible. Do not hide or obscure the browser's built-in audio player controls with CSS. They are the most reliably accessible controls available.
  • Make playback controls large enough to tap. Ensure all interactive areas meet a minimum touch target size of 44×44 px so the player is usable on mobile without precision pointing.
  • Design transcripts as readable body text. Format transcripts using normal typographic styles — paragraph text with adequate line spacing and contrast. Avoid presenting them as raw code or in a monospace font.

Dev guidelines for web

  • Use <audio controls> — do not hide native controls. The controls attribute exposes keyboard-accessible playback, volume, seek, and speed controls managed by the browser. Removing it requires fully rebuilding that interaction with ARIA and JavaScript.
  • Set preload="none" or preload="metadata". Avoid preloading the full audio file on page load. preload="metadata" loads only duration and track information; preload="none" defers all loading until the user presses play.
  • Never use the autoplay attribute. This attribute violates WCAG 1.4.2 (Audio Control) and is disorienting for screen reader users.
  • Provide a transcript via a linked <a> or inline <details>. Place the transcript link immediately adjacent to the player in the DOM. A <details><summary>Show transcript</summary>…</details> pattern below the player keeps the page compact while keeping the alternative reachable.
  • Provide multiple <source> formats for browser compatibility. Include at minimum an mp3 source and an ogg source so the element works across all major browsers: <source src="audio.mp3" type="audio/mpeg"> followed by <source src="audio.ogg" type="audio/ogg">.
  • Use <figure> and <figcaption> to provide context. Wrapping the player in a <figure> with a <figcaption> associates a visible caption with the audio element and gives screen reader users immediate context about what they are hearing.

Example code

<figure>
  <figcaption id="audio-label">
    Podcast Episode 1: Introduction to Accessibility
  </figcaption>
  <audio
    controls
    aria-labelledby="audio-label"
  >
    <source src="episode1.mp3" type="audio/mpeg" />
    <p>
      Your browser does not support HTML audio.
      <a href="episode1.mp3">Download the audio file</a>.
    </p>
  </audio>
  <details>
    <summary>Transcript</summary>
    <p>Welcome to Episode 1. Today we cover the basics of
    web accessibility and why it matters for all users.</p>
  </details>
</figure>
