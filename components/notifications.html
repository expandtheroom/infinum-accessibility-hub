Notifications & Alerts

Accessible notification and alert patterns using ARIA live regions and correct roles.

Design guidelines

UX guidelines

  • Indicate severity using color + icon + text together — never color alone. Every type (error, warning, success, info) must include a visible label such as "Error:" or "Success:" alongside the message (WCAG 1.4.1).
  • Use a consistent icon set for each severity level so users learn to recognize the meaning at a glance.
  • Position notifications consistently across the app (e.g. top-right or bottom-center). Inconsistent placement is disorienting for all users.
  • If a notification auto-dismisses, it must not contain content requiring user action. Provide adequate reading time and pause the dismiss timer when the notification receives keyboard focus or mouse hover.
  • Avoid auto-dismissing notifications that contain errors or warnings — keep them visible until the user explicitly dismisses them.

Visual design guidelines

  • The close button must be clearly visible with an adequate touch target (minimum 44×44 px) and must not be obscured by notification content.
  • When multiple notifications stack, ensure they are clearly separated and do not overlap important page content or each other.

Dev guidelines for web

  • Use role="alert" for errors and urgent warnings — this is implicitly aria-live="assertive" and interrupts the screen reader immediately.
  • Use role="status" for confirmations and informational updates — this is implicitly aria-live="polite" and waits for the current speech to finish.
  • Inject message text into a pre-existing live region — render the container empty on page load, then set its textContent via JavaScript. Do not destroy and recreate the container; screen readers may lose track of it.
  • Use aria-live="assertive" only for critical errors requiring immediate attention. Overuse is disruptive and erodes trust. Prefer aria-live="polite" in almost all cases.
  • Mark status icons as aria-hidden="true" when a visible severity label is already present in the text, to prevent double-announcing the type.
  • Dismissible notifications must include a labeled close button: <button aria-label="Dismiss notification">. On dismissal, return focus to a sensible location — not into a void.
  • Test live region announcements with real screen readers (NVDA, JAWS, VoiceOver) — automated tools cannot verify whether an announcement actually fired.

Example code

<!-- Polite live region: announced when user is idle -->
<div
  role="status"
  aria-live="polite"
  aria-atomic="true"
  id="notification-region"
>
  <!-- Messages injected here via JavaScript -->
</div>

<!-- Assertive live region: interrupts immediately (use sparingly) -->
<div
  role="alert"
  aria-live="assertive"
  aria-atomic="true"
>
  <strong>Error:</strong>
  Failed to save changes. Please try again.
</div>
