Our Baseline

Every project delivered by Infinum NA must meet WCAG 2.2 Level AA as a minimum. This is not a stretch goal — it is the floor.

Why WCAG 2.2 Level AA?

WCAG 2.2 Level AA represents the internationally recognised standard for digital accessibility. It is referenced in legislation across the United States, European Union, United Kingdom, Canada, and Australia, and is the benchmark used by enterprise and government clients when assessing supplier compliance. Meeting it by default protects our clients legally, widens the reach of what we build, and reflects our values as a team.

Level A covers the most critical barriers — failures at this level make content outright unusable for many people with disabilities. Level AA builds on this with requirements around contrast, text resizing, consistent navigation, and error prevention that meaningfully improve usability for a much broader population. Together they represent what we consider to be a professionally responsible minimum.

Level AAA criteria are not required on every project, but specific criteria should be pursued where the use case warrants it — for example, sign language video for products with a deaf user base, or enhanced contrast for medical or safety-critical interfaces. Discuss AAA applicability with the client during scoping.

What our baseline covers

WCAG 2.2 is organised around four principles — content must be Perceivable, Operable, Understandable, and Robust. The Level AA requirements span the following areas, with the following exceptions.

  • Infinum NA can't be held responsible for any content added to the CMS from a client - this could include images without alt text which would break guideline 1.1.1 or headings in the wrong order which would break 1.3.2.
  • Our baseline offering would not cover captioning or audio descriptions for any new or existing video content (Guidelines 1.2.1-1.2.3)
  • Accessibility for PDFs or other documents would not be part of the baseline
  • Third-party plugins or tools: while we can recommend the best accessible options, we are not responsible for their accessibility compliance.

What's covered

Perceivable

  • Text alternatives — all non-text content (images, icons, charts) has a text equivalent. Decorative elements are hidden from the screen reader.
  • Adaptable content — information and structure can be conveyed through different presentations without losing meaning. Content does not rely on shape, color, size, visual location, orientation, or sound alone.
  • Orientation — content does not restrict its view and operation to a single display orientation (e.g. portrait or landscape) unless essential.
  • Identify input purpose — the purpose of form fields collecting personal information can be programmatically determined, meaning that we use input autocomplete attributes when possible.
  • Color contrast — text — normal text meets a 4.5:1 contrast ratio against its background; large text (18pt or 14pt bold) meets 3:1.
  • Color contrast — UI components and graphics — interactive elements and meaningful graphics meet a 3:1 contrast ratio against adjacent colors.
  • Resize text — text can be resized up to 200% without loss of content or functionality.
  • Images of text — text is used instead of images of text wherever possible. Images of text have appropriate accessible descriptions
  • Reflow — content reflows to a single column at 320px wide (equivalent to 400% zoom) without requiring horizontal scrolling, except for content that requires two-dimensional layout.
  • Non-text contrast — focus indicators, form field boundaries, and meaningful graphical elements have sufficient contrast.
  • Text spacing — no content or functionality is lost when letter spacing, word spacing, line height, and paragraph spacing are adjusted within specified thresholds.

Operable

  • Keyboard accessible — all functionality is operable via keyboard alone, with no keyboard traps.
  • No keyboard trap — keyboard focus is never locked within a component unless it is a modal or intentional focus management, and there is always a way to exit.
  • Timing — users can turn off, adjust, or extend any time limits, unless the time limit is essential (e.g. a real-time auction).
  • Pause, stop, hide — moving, blinking, scrolling, or auto-updating content can be paused, stopped, or hidden.
  • Three flashes or below threshold — content does not flash more than three times per second, or flashing is below general and red flash thresholds.
  • Bypass blocks — a mechanism exists to skip past repeated content (e.g. a skip to main content link).
  • Page titled — every page has a descriptive, unique title.
  • Focus order — focus moves in an order that preserves meaning and operability.
  • Link purpose — the purpose of each link is clear from its text or its context.
  • Multiple ways — more than one way is available to locate a page within a set of pages (e.g. search and site map).
  • Headings and labels — headings and labels describe their topic or purpose.
  • Focus visible — any keyboard-operable component has a visible focus indicator with enough.
  • Focus appearance (new in 2.2) — the focus indicator meets minimum size and contrast requirements: at least a 2px outline with a 3:1 contrast ratio change between focused and unfocused states.
  • Pointer gestures — functionality that uses multipoint or path-based gestures (e.g. pinch, swipe) can also be operated with a single pointer.
  • Pointer cancellation — actions triggered on the down event can be cancelled or reversed.
  • Label in name — for UI components with visible labels, the accessible (programatic) name contains the visible label text.
  • Motion actuation — functionality triggered by device motion can also be operated through UI controls and can be disabled.
  • Dragging movements (new in 2.2) — any functionality that uses dragging can also be achieved without dragging using a single pointer.
  • Target size (new in 2.2) — interactive targets are at least 24×24 CSS pixels, or have sufficient spacing so that the total target area including spacing is at least 24×24 pixels.

Understandable

  • Language of page — the default human language of each page is programmatically determined.
  • Language of parts — the language of passages or phrases that differ from the page's default language is identified.
  • On focus — receiving focus does not trigger a change of context.
  • On input — changing a setting does not automatically trigger a change of context unless the user has been informed in advance.
  • Consistent navigation — navigation mechanisms that repeat across pages appear in the same relative order each time.
  • Consistent identification — components with the same function are identified consistently across pages.
  • Error identification — if an input error is detected, the item in error is identified and the error is described to the user in text.
  • Labels or instructions — labels or instructions are provided where content requires user input.
  • Error suggestion — if an input error is detected and suggestions for correction are known, they are provided to the user.
  • Error prevention for legal, financial, and data — submissions are reversible, checked for errors, or confirmed before finalising.
  • Accessible authentication (new in 2.2) — cognitive function tests (e.g. transcribing distorted text) are not required to authenticate unless an alternative is provided.
  • Redundant entry (new in 2.2) — information previously provided by the user within the same process is auto-populated or available for selection, and does not need to be re-entered.

Robust

  • Name, role, value — for all UI components, the name and role can be programmatically determined, and states, properties, and values that can be set by the user can be programmatically set. Changes to these are available to assistive technologies. We should always strive to use native components over custom.
  • Status messages — status messages can be programmatically determined without receiving focus so that assistive technologies can announce them to the user. Any toast notification or change of context results in an ARIA-live message.
