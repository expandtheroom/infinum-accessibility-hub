Way of Working

Accessibility is a shared responsibility. Here's how each discipline integrates it into their work.

Biz Dev

  • All projects should seek to meet WCAG 2.2 Level AA by default — this should be the baseline expectation set during scoping, not a bolt-on at the end. If a client explicitly descopes accessibility, this should be documented.
  • Include a baseline accessibility allocation in every project estimate — plan for [X] hours of accessibility consideration in design and [X] hours in development. This should be included as line items in our estimate templates.
  • Accessibility-specific engagements should be scoped separately:
    • Audit of an existing product — 30–40 hours. Covers identifying unique page templates, testing each against WCAG criteria (keyboard, screen reader, zoom, contrast, etc.), and producing a written report with prioritized findings.
    • Follow-up audit after remediation — 20–30 hours. Re-tests the issues identified in the original audit to verify fixes, and checks for any regressions.
    • VPAT (Voluntary Product Accessibility Template) — 10 hours in addition to the audit. Produces a formal conformance report in VPAT format, required by many enterprise and government clients.

Producers

  • Know the guidelines well enough to hold the team accountable — producers do not need to be accessibility experts, but they should be familiar with the expectations across UX, visual design, development, and QA outlined in this hub. This allows you to catch gaps early and advocate for the right outcomes when tradeoffs arise.
  • Make accessibility testing a scheduled, named phase — accessibility review must appear explicitly in the project schedule. It should not be absorbed into general QA time or left open-ended at the end of the project. Assign clear owners and define what "done" looks like (e.g. zero WCAG AA blockers, reviewed against this hub's component checklist).
  • Build accessibility into estimates and statements of work from the start — coordinate with Biz Dev to ensure that accessibility hours are included in the project scope. If a client removes accessibility from scope, document this explicitly and flag the legal and reputational risks.
  • Track accessibility issues and sign-off at each handoff gate — treat open accessibility issues like open bugs. At each handoff (UX to visual design, design to dev, dev to QA), confirm that the outgoing discipline has completed their accessibility responsibilities before the baton passes. Do not let issues accumulate until the end.
  • Raise accessibility blockers early with clients — if a client-directed design decision, third-party integration, or scope reduction creates an accessibility risk, flag it in writing as soon as it is identified. Decisions made late in a project are significantly harder and more expensive to remedy.

UX

  • Review the UX guidelines for the specific components you are designing — the components section of this hub contains interaction-specific guidance. Check it before finalising flows or handing off.
  • Consider accessibility use cases throughout the design process, not just at review — ask questions like:
    • Is this interaction cognitively taxing? Could it be simplified?
    • Does this require a complex gesture (e.g. multi-finger swipe, long press) that may be difficult for some users to perform?
    • Does this feature rely on unlabeled icons or color alone to convey meaning?
    • Is there a time limit on this interaction? Can it be extended or removed?
  • Annotate designs with accessibility-specific documentation before handoff — annotations reduce ambiguity and prevent developers from making incorrect assumptions. Include:
    • Whether each interactive element is a <button> or <a>
    • Focus order for complex layouts and modal flows
    • Accessible names for icon-only controls, images, and form fields
    • Any expected ARIA roles, states, or live region behavior

Visual Design

  • Review the visual design guidelines for the specific components you are designing — the components section of this hub contains visual-specific guidance for each component type.
  • Always test color contrast and document which combinations are accessible — for every color palette, produce a matrix showing which foreground/background pairings meet WCAG AA (4.5:1 for text, 3:1 for UI elements) and which do not. Figma plugins like Contrast Grid can automate this.
  • Always design a skip to main content link — it must be the first focusable element on every page and must be visible when focused. Design it as a styled component, not an afterthought.
  • Always design the focused state for every interactive element — focus indicators must be visible and meet WCAG 2.2 contrast requirements (3:1 against adjacent colors, minimum 2px outline). Do not leave focus styling to the browser default.
  • Consider producing a high-contrast and an enlarged-font variant — this does not need to cover every screen. A representative sample (e.g. home, a form, a data table) showing how the design holds up at 200% font size and with OS high-contrast mode enabled is valuable guidance for developers.
  • Never use color or shape alone to convey meaning — error states, status indicators, chart data, and required fields must always include a text label or icon in addition to color. This is a WCAG Level A requirement.

Dev

  • Review the dev guidelines for the specific components you are building — the components section of this hub contains implementation guidance for each component type, including correct HTML semantics, ARIA usage, and keyboard behavior.
  • Build with accessibility in mind from the start, not as a remediation step — retrofitting accessibility onto inaccessible markup is significantly more expensive than getting the semantics right first. Use native HTML elements wherever possible before reaching for ARIA.
  • Test for accessibility as you build, not only at the end — run axe or Lighthouse on each component as it is developed. Test keyboard navigation and basic screen reader behavior before submitting for review.
  • Ask designers when something is unclear — if a handoff is missing focus states, accessible names, or interaction annotations, ask before making assumptions. Incorrect implementations are harder to fix after QA.

QA

  • Identify unique page templates before beginning an audit — when auditing a product that was not built by our team, start by cataloguing the distinct templates (e.g. home, listing, detail, form, checkout). Test one representative instance of each rather than duplicating effort across identical pages.
  • Test each template across multiple assistive technologies and input methods:
    • Keyboard only — tab through every interactive element, verify visible focus indicators, confirm all functionality is reachable and operable without a mouse
    • Screen reader — test with at least two screen reader and browser combinations (e.g. NVDA + Chrome, VoiceOver + Safari) and verify that content, roles, states, and labels are announced correctly
    • Switch access on mobile — verify that interactive elements can be reached and activated using iOS Switch Control or Android Switch Access
  • Enlarge the font to 200% and check for content loss — increase the browser font size to 200% (not zoom, but text size) and verify that no content is clipped, overlapping, or lost. This tests WCAG 1.4.4 Resize Text.
  • Toggle light mode and dark mode on mobile — switch the OS appearance setting and verify that contrast, colors, and component states are correct in both modes, particularly for custom color schemes that may not follow the system palette.
